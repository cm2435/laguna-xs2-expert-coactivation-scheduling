# Pool + HF Rollout ACP Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Use `pool` as the coding-agent scaffold while keeping HF/PyTorch Laguna XS.2 as the model backend that generates tokens and captures token-aligned artifacts.

**Architecture:** `pool` owns the agent loop, repo workspace, shell/file/edit tools, permissions, and rollout lifecycle. A project-owned ACP server exposes a local HF/PyTorch Laguna agent to `pool`; every assistant token is generated inside our Python process, where we can log tokens and attach PyTorch hooks for activations.

**Tech Stack:** pool CLI, Agent Client Protocol (ACP), Python 3.11, PyTorch, Transformers, Laguna XS.2 BF16, JSONL/PT artifact storage, pytest.

---

## Executive Summary

Stage 1 proved that Laguna XS.2 BF16 can be loaded through HF/PyTorch on the B300 VM and can generate sane Python coding outputs using the official model-card prompt format.

Stage 2 should make those generations happen inside a realistic coding-agent scaffold. We do **not** want to rebuild a coding agent harness by hand. `pool` already does the painful pieces: workspace context, file reads, shell execution, edit application, permissions, non-interactive task execution, and trajectory/log management.

The boundary is:

```text
pool
  owns: agent loop, task scaffold, tools, repo state, observations

our HF ACP server
  owns: model loading, chat formatting, token generation, token logging,
        activation capture, run artifacts
```

This gives us the best of both worlds:

```text
pool exec
  -> ACP request to our local agent server
  -> HF/PyTorch Laguna generate(...)
  -> capture tokens and activations to disk
  -> response returns to pool
  -> pool executes tool actions and produces observations
  -> repeat
```

## Why Not Use Poolside's Hosted Backend?

Poolside's hosted backend is useful for creating realistic transcripts, but it cannot be the core training backend for densification.

We need:

- exact token ids generated by our checkpoint,
- PyTorch forward hooks,
- MoE block inputs and outputs,
- router metadata,
- gradients through surrogate layers later,
- reproducible checkpoint/module control.

Hosted Poolside generations do not expose those tensors. Therefore, every model call used for activation capture must run through our HF/PyTorch process.

The fallback use of Poolside's backend is transcript bootstrapping:

```text
pool + hosted backend
  -> save trajectory/logs
  -> replay prompts through HF/PyTorch Laguna
  -> capture activations
```

That is useful if ACP integration blocks, but it is not the preferred path.

## Target Control Flow

```text
1. Start HF Laguna ACP server
   uv run python scripts/run_hf_laguna_acp_server.py --config configs/hf_laguna_acp.yaml

2. Run pool against that server
   pool --agent-server "uv run python scripts/run_hf_laguna_acp_server.py --config configs/hf_laguna_acp.yaml"

3. Non-interactive rollout
   pool --agent-server "..." exec -p "Fix the failing test in this repo" -o json --unsafe-auto-allow

4. On each model turn
   pool sends session state / transcript over ACP
   server formats messages with tokenizer.apply_chat_template(..., enable_thinking=true)
   server calls model.generate(...)
   server writes token artifacts
   server optionally captures activations
   server returns assistant text to pool

5. Pool executes tools
   file reads, shell commands, edits, observations

6. Next turn repeats
```

The exact `pool --agent-server ... exec ...` syntax must be verified during the first spike. The README confirms that `pool --agent-server` can connect to arbitrary ACP servers, and `pool exec` can run non-interactively.

Pool should run inside disposable task sandboxes, not inside repo templates or the project source tree. Task manifests, repo templates, public tests, and hidden graders are defined in [RFC 007: Real Repo Task Environments](../rfcs/007-real-repo-task-environments.md).

```python
from __future__ import annotations

import json
from dataclasses import asdict, dataclass
from pathlib import Path

import torch


@dataclass(frozen=True)
class ModelCallMetadata:
    call_id: str
    task_id: str | None
    input_token_count: int
    generated_token_count: int
    activation_capture: str


def write_model_call_artifacts(
    call_dir: str | Path,
    messages: list[dict[str, str]],
    input_tokens: torch.Tensor,
    generated_tokens: torch.Tensor,
    generated_text: str,
    metadata: ModelCallMetadata,
) -> None:
    path = Path(call_dir)
    path.mkdir(parents=True, exist_ok=True)
    (path / "messages.json").write_text(json.dumps(messages, indent=2) + "\n", encoding="utf-8")
    (path / "generated_text.txt").write_text(generated_text, encoding="utf-8")
    (path / "metadata.json").write_text(
        json.dumps(asdict(metadata), indent=2) + "\n",
        encoding="utf-8",
    )
    torch.save(input_tokens.detach().cpu(), path / "input_tokens.pt")
    torch.save(generated_tokens.detach().cpu(), path / "generated_tokens.pt")
```

- [ ] **Step 1: Implement interface around tokenizer/model**

```python
from __future__ import annotations

from dataclasses import dataclass
from pathlib import Path

import torch

from densify.capture.token_artifacts import ModelCallMetadata, write_model_call_artifacts
from densify.config import GenerationConfig


@dataclass(frozen=True)
class HFAgentResponse:
    text: str
    call_id: str
    input_token_count: int
    generated_token_count: int


class HFLagunaAgent:
    def __init__(self, model, tokenizer, generation: GenerationConfig, run_dir: Path):
        self.model = model
        self.tokenizer = tokenizer
        self.generation = generation
        self.run_dir = run_dir
        self.call_index = 0

    def _next_call_id(self) -> str:
        self.call_index += 1
        return f"call_{self.call_index:06d}"

    @torch.inference_mode()
    def generate(self, messages: list[dict[str, str]], task_id: str | None = None) -> HFAgentResponse:
        call_id = self._next_call_id()
        device = next(self.model.parameters()).device
        encoded = self.tokenizer.apply_chat_template(
            messages,
            add_generation_prompt=True,
            return_tensors="pt",
            enable_thinking=self.generation.enable_thinking,
        )
        if isinstance(encoded, torch.Tensor):
            inputs = {"input_ids": encoded.to(device)}
        else:
            inputs = {key: value.to(device) for key, value in encoded.items()}

        input_ids = inputs["input_ids"]
        input_len = int(input_ids.shape[-1])
        kwargs = {
            **inputs,
            "max_new_tokens": self.generation.max_new_tokens,
            "do_sample": self.generation.do_sample,
            "pad_token_id": self.tokenizer.pad_token_id,
            "eos_token_id": self.tokenizer.eos_token_id,
        }
        if self.generation.do_sample:
            kwargs["temperature"] = self.generation.temperature
            kwargs["top_p"] = self.generation.top_p
            if self.generation.top_k is not None:
                kwargs["top_k"] = self.generation.top_k

        output_ids = self.model.generate(**kwargs)
        generated_tokens = output_ids[0, input_len:]
        text = self.tokenizer.decode(generated_tokens, skip_special_tokens=True)

        write_model_call_artifacts(
            self.run_dir / "model_calls" / call_id,
            messages=messages,
            input_tokens=input_ids,
            generated_tokens=generated_tokens,
            generated_text=text,
            metadata=ModelCallMetadata(
                call_id=call_id,
                task_id=task_id,
                input_token_count=input_len,
                generated_token_count=int(generated_tokens.numel()),
                activation_capture="tokens_only",
            ),
        )
        return HFAgentResponse(
            text=text,
            call_id=call_id,
            input_token_count=input_len,
            generated_token_count=int(generated_tokens.numel()),
        )
```

## Risks And Decisions

### Risk: ACP Server Is More Work Than Expected

Fallback:

```text
pool default backend
  -> collect transcript/logs
  -> replay transcript prompts through HF Laguna
  -> capture activations
```

This loses "HF generated the actual tool actions" but still gives realistic scaffold-shaped contexts.

### Risk: Pool Requires Rich Tool-Call Semantics

If Laguna emits plain code/prose rather than structured pool actions, pool may not execute useful tools.

Mitigation:

- First make pool accept a plain assistant answer.
- Then inspect Poolside action format from pool logs.
- Add prompt instructions or response adapter only if needed.

### Risk: Activation Storage Explodes

Start with:

```text
tokens only
one model call
one sparse layer
one short task
```

Then scale layers and turns.
