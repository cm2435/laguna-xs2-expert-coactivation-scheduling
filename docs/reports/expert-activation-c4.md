**Model:** `poolside/Laguna-XS.2`  
**Corpus:** allenai/c4 (en) excerpt — 400 docs, 161,932 tokens (seq_len ≤ 1024)  
**Routing:** top-8 of 256 experts per token, across 39 sparse-MoE layers (layer 0 is dense).

Generated by forward-pass router introspection (hooks on every `LagunaTopKRouter`). Inspired by the RADLADS distillation framing (arXiv:2505.03005): measure what the sparse model actually uses before transferring it to a denser surrogate.

## 6. Implications for MoE→dense densification

- With only ~**158 effective experts** per layer (of 256) on general C4 text, the routed FFN behaves far denser than its 256-way capacity suggests — a dense surrogate MLP per layer has a realistic target to reconstruct.
- Load Gini ≈ 0.53 means usage is skewed: a dense surrogate should be fit preferentially on the high-traffic experts; low-traffic experts contribute little to C4 activations.
- The coactivation structure (§4) identifies experts that habitually fire together — natural groupings for block-wise densification or co-scheduling, matching this repo's *expert-coactivation-scheduling* premise.
- Coverage is domain-dependent: C4 is general web text; code/math prompts would light up a different expert subset. A production densification set should mix domains so the surrogate isn't overfit to C4's active experts.
